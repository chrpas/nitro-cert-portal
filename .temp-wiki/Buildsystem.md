Nitro5 is using a buildsystem that is powered by [NodeJS](https://nodejs.org/en/). It makes use of a npm package called [@avensia/nitro5-buildsystem](https://github.com/avensia/nitro5-packages/tree/nitro5-develop/packages/nitro5-buildsystem) where common tasks are defined. Under the hoods, another Avensia npm package, [garn](https://github.com/avensia/garn) is used.

You invoke the buildsystem by running `nitro` in a command prompt, followed by a task name.:

## Common tasks
- `nitro build` - builds both backend and frontend
- `nitro backend:build` - builds backend + invokes [tstypegen](https://github.com/avensia/nitro5/wiki/tstypegen) is also invoked on `Avensia.Common` (same as building in Visual Studio)
- `nitro frontend:build` - builds frontend (using tsc)
- `nitro frontend:watch` - watches for modifications to frontend files and rebuilds the frontend code instantly. This is handy when you work with frontend code to speed up compilation.
- `nitro start-site` - starts the site. It will stop all running iis websites on port 443 and then run `dotnet run --project src/Avensia.Shop/Avensia.Shop.csproj`.

> Tip: When the buildsystem is started, [yarn](https://yarnpkg.com/) performs an integrity check - much like `nuget restore`. This check really only needs to run when frontend packages have been updated/added. To skip this check in everyday work, you can add parameter `--skip-yarn` or `-sy` to your build command.

In addition to build support, the buildsystem also contains a wealth of tasks that can be helpful to streamline and automate the developer experience:

## Setup
- **setup-dev-env**, see [How to setup Nitro5 dev environment](https://github.com/avensia/nitro5/wiki/How-to-setup-Nitro5-dev-environment)
- **restore-env** will ask if you want to download and restore databases and assets, and if you do, it will do so using tasks `download-backups` and `download-assets` (see below)

And also:
- **setup-esales** is run during `setup-dev-env` and run tasks `validate-esales`, `download-esales`, `start-esales`, `define-esales-cluster`.
- **validate-esales** is run during `setup-dev-env` and checks that `java.exe` is in your PATH
- **download-esales** is run during `setup-dev-env` and downloads esales to your `/esales` folder
- **define-esales-cluster** is run during `setup-dev-env` and defines an eSales cluster for your local environment
- **setup-certificates** is run during `setup-dev-env` and sets up SSL certificates for your local environment
- **check-if-admin** is run during `setup-dev-env` and checks that you ran `nitro.cmd` as administrator
- **download-backups** will ask if you want to download database backups, and if so, runs the `download-backups-from-storage` task
- **download-backups-from-storage** will download the database backups from the Avensia FTP 
- **restore-databases** (or its synonym **restore-backups**) restores the cms and commerce `.bak` files in your `/downloads` folder
- **download-assets** will ask if you want to download assets, and if so, runs the `download-assets-from-storage` task
- **download-assets-from-storage** will download assets from the Avensia FTP
- **create-debug-configs** will be run during `setup-dev-env` and creates the `buildsystem/config.local.json` file
- **validate-debug-configs** does nothing, always returns true.

## Misc tools during dev
- **run-migrations** will first run task `start-esales` to verify that esales is running (if your contentindex type is esales), and will then run all idempotent migrations followed by all not yet run migrations
- **create-migration** will create a new class inheriting from `IMigration` in the correct folder for the current year in `Avensia.Common/Migrations`
- **sync-env** will run tasks `start-esales`, `run-migrations`, `full-promo-cache`, `full-index`
- **fix-env** will run tasks `sync-env` and `frontend:build` in parallell
- **start-esales** will first stop esales if it is already running, and then start it
- **stop-esales** will stop esales 
- frontend:prettier
- frontend:analyze
- frontend:typecheck
- frontend:dev-server
- snapshot-create
- snapshot-restore
- snapshot-delete
- **update-scope** will update all Scope  packages to the latest version (including nuget packages `Avensia.Nitro5.Scope`, `Avensia.Nitro5.Scope.Mvc`, `Avensia.Nitro5.Scope.EpiServer` and npm packages `@avensia/nitro5-scope`, `@avensia/nitro5-scope-episerver`, `@avensia/nitro5-buildsystem`). Use this task to not forget any package when updating Scope.
- **cleanup-databases** will clean a database and delete users, orders etc. Use when copying a database from production to dev. See https://github.com/avensia/nitro5-packages/pull/158
- **add-translation** will add a new translation to all your language files, and also automatically translate it using Google Translate. See https://github.com/avensia/nitro5-packages/pull/294

## Starting new project based on Nitro
- **setup-new-project** sets up the very first developer environment for a new project based on Nitro. Is used during "sprint zero" of a new project. See [How to start a new project based on Nitro5](https://github.com/avensia/nitro5/wiki/How-to-start-a-new-project-based-on-Nitro5)
- **validate-nitro-fixes** will find all remaining NITROFIX comments in solution. This will also fail the build on TC if any critical nitrofixes remains.

And also:
- **create-databases** is run during `setup-new-project` and will create empty cms and commerce databases using `.sql` files embedded in the Episerver nuget packages
- **validate-databases** is run during `setup-new-project` and will check that `sqlcmd.exe` is in your PATH
