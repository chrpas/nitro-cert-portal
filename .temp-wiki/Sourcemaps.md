Source maps is a way to access readable Javascript code even after it's been minified into bundles. This helps us locate frontend-related exceptions easier by **providing a stack trace** to the error. 

![image](https://user-images.githubusercontent.com/6929325/176145720-169f38e7-acc8-4493-8b14-15fb90ab9550.png)
*Stacktrace enabled in Raygun via source maps*

Source maps are generated by webpack during build. There is a naming convention for source maps, which is the bundled filename with a `.map` extension, e.g. `0.chunk.a66538314f69cc032e97.js.map`. You'll find these files in the `assets` folder of the shop.

## Raygun support for source maps
Nitro's buildsystem will automatically upload Javascript source maps to Raygun, see [Setting up Raygun](https://github.com/avensia/nitro5/wiki/Setting-up-Raygun) on how to set it up.

Source maps are **uploaded** every time the "Publish Release" build is triggered, which in the default setup means **every time we tag a version in the project**. More specifically, the call chain goes like this: `scope.cmd publish-and-prepare-deploy` => `publish` => `package` => `frontend:send-sourcemaps-to-raygun` => `upload-to-raygun`.


## Chrome support for source maps
Chrome will automatically try to access Javascript source maps if you have developer tools open. But we don't want to provide the source maps to anyone, which is why we are providing empty source maps via the `SourceMapsMiddleware` middleware. To see the real source maps, log in as Administrator.
