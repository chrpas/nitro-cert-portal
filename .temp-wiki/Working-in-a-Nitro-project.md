## Nitro projects

A typical Nitro projects goes through these phases:

1. **Detailing phase**. This is where we do a GAP-analysis (comparing customer's reqiurements with what's in Nitro) and also design the UX.
2. **Project startup phase**. Then we start the project by going into what we usually call “sprint 0”. This is usually the Solution Architect cloning Nitro into a customer project Github repo, and making some startup configurations for the project (See [Starting a new Nitro project](https://github.com/avensia/nitro5/wiki/Starting-a-new-Nitro5-project))
3. **Development phase**. Then the actual development starts with a first sprint. Development is usually 3-6 months but could be shorter and could be longer. It is usually in 2 weeks sprints. This is where we do all the development, where we adapt what needs to be adapted in Nitro and implement all customer specific features. One thing that we always need to adapt is the UX. 
4. **UAT phase**. When we are ready to go-live, there is usually first a few weeks of User Acceptance Tests made by the customer before a final go-live-decision is taken.
5. The actual **go-live**. (See [Go-live checklist for Nitro projects](https://github.com/avensia/nitro5/wiki/Go-live-checklist))
6. Two weeks of **hypercare** meaning the entire development team is still fully available to quickly fix issues discovered in production after go-live.
7. Project goes either into **maintenance mode** or just keeps on developing sprint after sprint, deploying new features to production continously.

## Working as a developer in a Nitro project

Working in a Nitro project has a few pretty nice benefits:

- We are a large developer community – you are not alone! If you get stuck on something, there is a good chance that someone else already got stuck on this very same problem before you and can help you. Also you can help others, this is a great opportunity to connect to developers beyond your specific project.
- It's easy to switch between different projects – everything from code base to project setup is similar in all Nitro projects.
- You have a great possibility to contribute and improve – you can do improvements that have a major impact, not only for your specific customer, but for all current and future Nitro customers.
- We have a well thought out way of developing high performance sites. Many basic constructions are already in place, so you can focus on what’s important for your specific customer.

But it also has a few downsides:

- It's a complex solution. We know that onboarding can be frustrating (and that is why we are writing these docs to help you!).
- Especially onboarding for frontend can be difficult. For example, the [Glitz](https://github.com/frenic/glitz) package is new to (more or less) all frontenders, even senior ones.
